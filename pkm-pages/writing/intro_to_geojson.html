GeoJSON is a common format throughout the web for representing geographic data, but it can seem daunting at first. This is a short overview of GeoJSON and its structure to help make more sense of it.

As the name implies, GeoJSON is just a format for representing various types of geographic data using JSON. There are six types of geometries that can be represented, and a few different containers that group them together or provide extra information.

Points and Positions

The simplest geometry is the Point. Points are commonly used to indicate places of interest and other markers on maps where you just want to show the location, and don't need to an area or boundary.

A point contains just a single position. In GeoJSON, a position is an array with two numbers: the longitude and the latitude, in that order. You may be accustomed to the more common "latitude, longitude" ordering. This is a common source of confusion when getting started, so it helps to think in terms of "X, Y" coordinates instead.

A position array may have a third value to represent elevation, but this isn't commonly seen.

Each GeoJSON geometry has a type field indicating what type of geometry it is, and a coordinates field. A point might look like this:

{
  "type": "Point",
  "coordinates": [137.2, -32]
}

You may also place multiple points into a single geometry using the MultiPoint type. In this case, the coordinates are an array of points.

{
  "type": "MultiPoint",
  "coordinates": [
    [137.2, -32],
    [123.1, -30]
  ]
}

MultiPoint should only be used when the set of points are conceivably actually part of the same object. There are better ways to represent multiple, independent points, which I'll cover later.

Making Geometries More Useful

A geometry, on its own, is just a shape, with no way to tell anything more useful about it. Yet you almost always want to add some kind of information to it. GeoJSON solves this with the Feature type, which wraps a geometry and lets you add properties to it.

{
  "type": "Feature",
  "id": "marker-001",
  "properties": {
    "name": "My favorite place",
    "icon": "star",
    "rating": "a billion"
  },
  "geometry": {
    "type": "Point",
    "coordinates": [137.2, -32]
  }
}

The id property is optional but can be useful for linking a Feature back to a particular object in a database, or something similar.

There is no defined set of valid data for properties here -- GeoJSON provides a designated place to put the properties but you can put anything you want inside that object.

When you want to group multiple Features together, you can use a FeatureCollection, which just contains an array of Features.

{
  "type": "FeatureCollection",
  "features": [
    { type: "Feature", properties: { }, geometry: { } },
    { type: "Feature", properties: { }, geometry: { } }
  ]
}

GeoJSON also provides the GeometryCollection type, which is similar but just holds a list of geometries of any type. In most places where a GeometryCollection could be used, one of the "Multi" objects is a better fit, so this doesn't come up too often, but it can be useful on occasion.

{
  "type": "GeometryCollection",
  "geometries": [
    { "type": "Point", coordinates: [1, 2] },
    { "type": "LineString", coordinates: [3, 4] }
  ]
}

Lines

The LineString type is useful when drawing lines on the map, such as when suggesting a route between to places. It contains a set of points that connect together, in order, to form a line. As with MultiPoint, it contains an array of positions, but a valid LineString must contain at least two positions.

{
  "type": "LineString",
  "coordinates": [
    [137.2, -32],
    [123.1, -30],
    [125.2, -31.2]
  ]
}

It follows that a MultiLineString represents a set of related lines. Its coordinates are an array of LineString coordinates, with each array member drawing a single line. Although the lines are considered part of the same geometry, they are not connected together. The example below would appear as two separate lines, with the second line 10 degrees above the first.

{
  "type": "MultiLineString",
  "coordinates": [
    [
      [137.2, -32],
      [123.1, -30],
      [125.2, -31.2]
    ],
    [
      [137.2, -22],
      [123.1, -20],
      [125.2, -21.2]
    ]
  ]
}

Polygons

Polygons are one of the most used geometries used in GeoJSON, but they bring in additional complexity. Polygons are commonly found when drawing boundaries around geographic regions or other areas. There is no limit on the number of points in a polygon. It can be as simple as a triangle or complex enough to follow the exact contours of a shoreline.

Polygons do come with a few restrictions. Unlike a LineString, a Polygon must be explicitly closed. The first and last points in the Polygon must be identical for it to be valid.

Furthermore, the points in a polygon must be in counterclockwise order. This rule makes it simpler for renderers to figure out which side of the polygon is the outside vs. the inside, without needing to do additional calculations.

The first LineString above would be represented like this:

{
  "type": "Polygon",
  "coordinates": [
    [
	  [137.2, -32],
      [123.1, -30],
      [125.2, -31.2],
      [137.2, -32]
    ]
  ]
}

You'll note that although there's just a single LineString here, the array nesting looks more like a MultiLineString. This is because Polygons allow you to specify holes inside the polygon as well. The inner holes need to have their points represented in clockwise order, which again helps renderers to determine which side of the edge is inside the polygon.

This Polygon is a square with two smaller square holes inside it.

{
  "type": "Polygon",
  "coordinates": [
    [
      [ 125, -23 ],
      [ 125, -28 ],
      [ 130, -28 ],
      [ 130, -23 ],
      [ 125, -23 ]
    ],
    [
      [126, -24],
      [127, -24],
      [127, -25],
      [126, -25],
      [126, -24]         
    ],
    [
      [128, -26],
      [129, -26],
      [129, -25],
      [128, -25],
      [128, -26]
    ]
  ]
}

It may feel daunting to have to ensure that your polygon's points are in the correct order, but geometry libraries generally handle this for you, and they often provider functions such as turf's rewind or PostGIS's ST_ForcePolygonCCW, which will take a polygon and "fix" the point order so that the outer line string is counterclockwise and the inner ones are clockwise.

MultiPolygons

By now you can probably guess what a MultiPolygon is. These are very useful in representing geographic regions such as the state of Michigan or anything with islands, which can't be represented as a single contiguous shape.

It starts to get hard to read with all the nested arrays, but the coordinates for a MultiPolygon is just an array of Polygon coordinates. This example shows a MultiPolygon that contains two polygons, one a simple polygon and the other with some holes.

{
  "type": "MultiPolygon",
  "coordinates": [
    [
      [
	    [137.2, -32],
        [123.1, -30],
        [125.2, -31.2],
        [137.2, -32]
      ]
    ],
    [
      [
        [ 125, -23 ],
        [ 125, -28 ],
        [ 130, -28 ],
        [ 130, -23 ],
        [ 125, -23 ]
      ],
      [
        [126, -24],
        [127, -24],
        [127, -25],
        [126, -25],
        [126, -24]         
      ],
      [
        [128, -26],
        [129, -26],
        [129, -25],
        [128, -25],
        [128, -26]
      ]
    ]
  ]
}

Summing Up
