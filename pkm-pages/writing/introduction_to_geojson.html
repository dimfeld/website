GeoJSON is a common format throughout the web for representing geographic data, and almost every map-based application uses it. But it can seem daunting at first.

This is a short overview of GeoJSON and its structure to help make more sense of it. In later articles in this series, I'll cover more interesting things you can do with geographic data and how to actually work with it, but this will provide a good foundation to build upon.

As the name implies, GeoJSON is just a format for representing geographic data using JSON. There are six types of geometries that can be represented, and a few different containers that group them together or provide extra information.

All images below are from geojson.io, a great tool for visualizing GeoJSON.

Points and Positions

The simplest geometry is the Point. Points are commonly used to indicate places of interest and other markers on maps where you just want to show the location, and don't need an area or boundary.

A Point contains just a single position. In GeoJSON, a position is an array with two numbers: the longitude and the latitude, in that order.

You may be accustomed to the more common "latitude first, longitude second" ordering. This is a common source of confusion when getting started, so it helps to think in terms of "X, Y" coordinates instead.

A position array may have a third value to represent elevation, but this isn't commonly seen.

Every GeoJSON geometry has a type field indicating what type of geometry it is, and a coordinates field. A point representing a random location in southern Australia looks like this:

{
  "type": "Point",
  "coordinates": [137.2, -32]
}

You may also place multiple points into a single geometry using the MultiPoint type. In this case, the coordinates are an array of points.

{
  "type": "MultiPoint",
  "coordinates": [
    [137.2, -32],
    [133.1, -30]
  ]
}

MultiPoint should only be used when the set of points are conceivably actually part of the same object. There are better ways to represent multiple, independent points, which I'll cover next.

Lines

The LineString type is useful when drawing lines on the map, such as when suggesting a route between two places. It contains a set of points that join together, connect-the-dots style, to form a line. As with MultiPoint, it contains an array of positions, but a valid LineString must contain at least two positions.

{
  "type": "LineString",
  "coordinates": [
    [ -46.64, -23.577 ],
    [ -43.2, -22.9 ],
    [ -43.94, -19.89 ]
  ]
}

GeoJSON also providers the MultiLineString type, which represents a set of related lines. Its coordinates are an array of LineString coordinates, with each array member drawing a single line.

{
  "type": "MultiLineString",
  "coordinates": [
    [
      [ -46.64, -23.577 ],
	  [ -43.2, -22.9 ],
      [ -43.94, -19.89 ]
    ],
    [
      [-48.18, -21.77],
      [-47.46, -20.58],
      [-46.57, -21.78]
    ]
  ]
}

Although the lines are considered part of the same geometry, they are not connected together. The example above appears as two separate lines in Brazil.

Making Geometries More Useful

A geometry on its own is just a shape, with no way to tell anything more useful about it. You almost always want to add some kind of information to it, and GeoJSON solves this with the Feature type. A Feature wraps a geometry and allows you add any metadata you want.

{
  "type": "Feature",
  "id": "marker-001",
  "properties": {
    "name": "My favorite place",
    "icon": "star",
    "rating": "a billion"
  },
  "geometry": {
    "type": "LineString",
    "coordinates": [
      [ -46.64, -23.577 ],
      [ -43.2, -22.9 ],
      [ -43.94, -19.89 ]
    ]
  }
}

The id property is optional but can be useful for linking a Feature back to a particular object in a database, or something similar.

There is no defined schema for properties here &emdash; you can put any JSON object inside there.

Groups of Objects

When you want to group multiple Features together, you can use a FeatureCollection, which just contains an array of Features.

{
  "type": "FeatureCollection",
  "features": [
    { 
      "type": "Feature", 
      "properties": { 
        "name":"my house"
      }, 
      "geometry": { 
      	"type": "Point",
        "coordinates": [ 1, 2 ]
      } 
    },
    // And so on....
    { "type": "Feature", "properties": { }, "geometry": { } }
  ]
}

GeoJSON also provides the GeometryCollection type, which is similar but holds a list of geometries of any type. You probably won't encounter GeometryCollection often. Usually , one of the "Multi" objects or a FeatureCollection is a better fit, but it's good to know about.

{
  "type": "GeometryCollection",
  "geometries": [
    { "type": "Point", "coordinates": [0, 51.5] },
    { 
      "type": "LineString",
      "coordinates": [
        [0, 51.5],
        [1.55, 50.7],
        [2.35, 48.85]
      ]
    }
  ]
}

Polygons

Polygons are one of the most used geometries used in GeoJSON, but they bring in additional complexity. Polygons are commonly found when drawing boundaries around geographic regions or other areas. There is no limit on the number of points in a polygon. It can be as simple as a triangle or complex enough to follow the exact contours of a shoreline.

Polygons do come with a few restrictions. Unlike a LineString, a Polygon must be explicitly closed. The first and last points in the Polygon must be identical for it to be valid.

Furthermore, the points in a polygon must be in counterclockwise order. This rule makes it simpler for renderers to figure out which side of the polygon is the outside vs. the inside, without needing to do additional calculations.

The first LineString above would be represented like this:

{
  "type": "Polygon",
  "coordinates": [
    [
	  [137.2, -32],
      [123.1, -30],
      [125.2, -31.2],
      [137.2, -32]
    ]
  ]
}

You'll note that although there's just a single LineString here, the array nesting looks more like a MultiLineString. This is because Polygons allow you to specify holes inside the polygon as well. The inner holes need to have their points represented in clockwise order, which again helps renderers to determine which side of the edge is inside the polygon.

This Polygon is a square with two smaller square holes inside it.

{
  "type": "Polygon",
  "coordinates": [
    [
      [ 125, -23 ],
      [ 125, -28 ],
      [ 130, -28 ],
      [ 130, -23 ],
      [ 125, -23 ]
    ],
    [
      [126, -24],
      [127, -24],
      [127, -25],
      [126, -25],
      [126, -24]         
    ],
    [
      [128, -26],
      [129, -26],
      [129, -25],
      [128, -25],
      [128, -26]
    ]
  ]
}

It may feel daunting to have to ensure that your polygon's points are in the correct order, but geometry libraries generally handle this for you, and they often provider functions such as turf's rewind or PostGIS's ST_ForcePolygonCCW, which will take a polygon and "fix" the point order so that the outer line string is counterclockwise and the inner ones are clockwise.

MultiPolygons

By now you can probably guess what a MultiPolygon is. These are very useful in representing geographic regions such as the state of Michigan or anything with islands, which can't be represented as a single contiguous shape.

It starts to get hard to read with all the nested arrays, but the coordinates for a MultiPolygon is just an array of Polygon coordinates. This example shows a MultiPolygon that contains two polygons, one a simple polygon and the other with some holes.

{
  "type": "MultiPolygon",
  "coordinates": [
    [
      [
	    [137.2, -32],
        [123.1, -30],
        [125.2, -31.2],
        [137.2, -32]
      ]
    ],
    [
      [
        [ 125, -23 ],
        [ 125, -28 ],
        [ 130, -28 ],
        [ 130, -23 ],
        [ 125, -23 ]
      ],
      [
        [126, -24],
        [127, -24],
        [127, -25],
        [126, -25],
        [126, -24]         
      ],
      [
        [128, -26],
        [129, -26],
        [129, -25],
        [128, -25],
        [128, -26]
      ]
    ]
  ]
}
